# WebAgentFlow Validation Site

This repository contains a standalone product-like validation site for WebAgentFlow.

It is not the WebAgentFlow product UI.
It is not an internal deterministic fixture site.
It is not a compatibility mirror of `apps/product-test-site`.

The local catalog page lists available validation surfaces:

```text
http://127.0.0.1:5177/
```

Current validation surfaces:

```text
http://127.0.0.1:5177/inventory
http://127.0.0.1:5177/users
```

This repository migrates product validation scenario intent from the historical
`apps/product-test-site`, but it does not preserve legacy routes such as
`/items`, `/workspace-login`, or `/workspace-home`.

## Repository Role

`WebAgentFlow-Validation-Site` is an external product-level validation target.
It gives WebAgentFlow a realistic business page for black-box browser operation
checks without colocating the site implementation with WebAgentFlow runtime code,
prompts, or eval runners.

The catalog currently links two validation surfaces:

- **Inventory Management** supports safe inventory operations:
  - create an inventory item in a dialog
  - search inventory items
  - edit an inventory item in a dialog
  - download the visible inventory table as CSV
- **User Directory** mirrors the fixture-site `/users` list page shape:
  - search by name, email, role, status, date, region, month, and department
  - open row-level user details
  - exercise no-match list states


## Boundary

The WebAgentFlow main repository may know only redacted provider output:

- opaque target reference
- opaque case reference
- aggregate status
- provider gate names
- redacted result artifacts

The WebAgentFlow main repository should not know:

- target URL, local port, or route
- private scenario details
- private user intent text
- expected behavior for this private target
- forbidden behavior details for this private target
- page source code
- component names
- `data-testid` values
- DOM selectors
- seed copy used by this repository
- internal state shape
- implementation-specific business logic

## Development

```bash
pnpm install
pnpm dev
```

`pnpm dev` starts both services:

- frontend: `http://127.0.0.1:5177`
- validation API: `http://127.0.0.1:8003`

Default local catalog and target:

```text
http://127.0.0.1:5177/
http://127.0.0.1:5177/inventory
http://127.0.0.1:5177/users
```

API endpoints used by the pages:

```text
GET  /api/inventory
POST /api/inventory
PUT  /api/inventory/:id
GET  /api/inventory/export.csv

GET  /api/users
GET  /api/users/meta/options
GET  /api/users/:id
```

LAN development target:

```bash
pnpm dev:lan
```

## Build

```bash
pnpm build
```

`pnpm-lock.yaml` is generated by `pnpm install`; do not hand-write it.

## Redacted Provider Summary

Private validation cases and raw artifacts stay in this repository. The only
artifact intended for WebAgentFlow consumption is:

```text
evals/artifacts/latest/provider-summary.redacted.json
```

Create or update a redacted summary:

```bash
pnpm run eval:summary -- --status UNVERIFIED --command-id summary-only
```
